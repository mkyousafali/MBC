# Database Instructions

## Creating Tables and RPC Functions

When creating database tables or RPC (PostgreSQL functions), follow this process:

1. **Load credentials from `.env`** — use `dotenv` or read `process.env.DATABASE_URL`. The connection string is stored in the `DATABASE_URL` environment variable.

2. **Create a script file** (e.g., `migrate.cjs` or `db-setup.cjs`) that:
   - Loads the connection string from `.env` using `require('dotenv').config()`
   - Connects via `pg` Client
   - Runs the SQL to create tables or functions
   - Logs success/failure
   - Closes the connection

3. **Example pattern:**
```js
require('dotenv').config();
const { Client } = require('pg');

const client = new Client({ connectionString: process.env.DATABASE_URL });

(async () => {
  await client.connect();

  // Create table
  await client.query(`
    CREATE TABLE IF NOT EXISTS my_table (
      id serial PRIMARY KEY,
      name text NOT NULL,
      created_at timestamptz DEFAULT now()
    )
  `);

  // Create RPC function
  await client.query(`
    CREATE OR REPLACE FUNCTION my_function(param text)
    RETURNS text
    LANGUAGE plpgsql
    AS $$
    BEGIN
      RETURN 'Result: ' || param;
    END;
    $$
  `);

  await client.end();
})();
```

## Security Rules — NEVER Commit Secrets

- **NEVER hardcode** database URLs, passwords, API keys, or tokens in script files.
- **ALWAYS load secrets from `.env`** using `process.env`.
- **Verify `.gitignore`** includes `.env` before every push.
- Scripts must contain **zero secrets** — only references to environment variables.
- If a script is generated with inline credentials, **refuse to commit it** and refactor to use `.env` first.
- Before any `git push`, confirm no secrets exist in staged files.

## File Naming

- Use `.cjs` extension for database scripts (project uses `"type": "module"` in package.json).
- Prefix migration scripts with a number or date for ordering (e.g., `001-create-tables.cjs`).

## Adding a New Sidebar Button (with Permission System)

When adding a new navigable button to the sidebar, follow **all 4 steps**:

### Step 1: Register the resource in the database

Create a migration (`.cjs`) that inserts into `app_resources`:

```sql
INSERT INTO app_resources (resource_key, main_section, sub_section, button_name, icon, component, display_order)
VALUES ('settings.operations.inventory', 'Settings', 'Operations', 'Inventory', '📦', 'InventoryWindow', 40)
ON CONFLICT (resource_key) DO UPDATE SET
  main_section = EXCLUDED.main_section, sub_section = EXCLUDED.sub_section,
  button_name = EXCLUDED.button_name, icon = EXCLUDED.icon,
  component = EXCLUDED.component, display_order = EXCLUDED.display_order;
```

The `resource_key` format is `section.subsection.name` (lowercase, dot-separated).

### Step 2: Add to sidebar `navItems` array

In `src/lib/components/desktop-team/DesktopSidebar.svelte`, add an entry to the `navItems` array:

```typescript
{ key: 'settings.operations.inventory', section: 'Settings', sub: 'Operations', id: 'inventory', title: 'Inventory', icon: '📦', component: 'InventoryWindow' }
```

The sidebar automatically filters items by permissions — only buttons where the user has `can_view = true` are shown. Empty sections/sub-sections are hidden. Super Admin sees everything.

### Step 3: Register the component in the layout

In `src/routes/team/desktop/+layout.svelte`:

```typescript
import InventoryWindow from '$lib/components/desktop-team/windows/InventoryWindow.svelte';
const componentMap = { ..., InventoryWindow };
```

Also add it to `src/routes/team/desktop/window/[id]/+page.svelte` for pop-out support.

### Step 4: Enforce permissions inside the window component

In the new window component, subscribe to `userPermissions` reactively:

```typescript
import { teamUser, userPermissions } from '$lib/stores/auth';

const RESOURCE_KEY = 'settings.operations.inventory';
let isSA = $state(false);
let permAdd = $state(false);
let permEdit = $state(false);
let permDelete = $state(false);
teamUser.subscribe(u => { isSA = u?.is_super_admin === true; });
userPermissions.subscribe(p => {
  const r = p.find((x: any) => x.resource_key === RESOURCE_KEY);
  permAdd = isSA || (r?.can_add ?? false);
  permEdit = isSA || (r?.can_edit ?? false);
  permDelete = isSA || (r?.can_delete ?? false);
});
```

Then use `{#if permAdd}`, `{#if permEdit}`, `{#if permDelete}` in the template to show/hide action buttons.

**Important:** Do NOT use the `canAdd()`, `canEdit()`, `canDelete()` functions from the auth store in templates — they use `get()` which is not reactive in Svelte 5. Always use the reactive `$state` subscription pattern above.

### Permission Rules

- **View**: User can see the sidebar button and open the page (read-only).
- **Add**: User can create records. Automatically requires View.
- **Edit**: User can edit records. Automatically requires View.
- **Delete**: User can delete records. Automatically requires View.
- Disabling View in the permissions UI automatically disables Add/Edit/Delete.
- Super Admin (`is_super_admin: true`) bypasses all permission checks.
- Permissions are refreshed on page load and window focus (no logout needed).

## Audit Log System

Every user action (including Super Admin) is recorded in the `audit_logs` table. The utility is at `src/lib/utils/audit.ts`.

### How it works

- `writeAuditLog(params)` — writes an entry. Automatically captures the current user from the `teamUser` store.
- `diffChanges(before, after, fields?)` — compares two plain objects and returns an array of `{ field, from, to }`.
- Window opens are logged automatically via `openWindow()` in `src/lib/stores/windows.ts`.

### Integrating audit logs into a NEW window

After creating your window component and registering it (Steps 1–4 above), add audit logging:

```typescript
import { writeAuditLog, diffChanges } from '$lib/utils/audit';
```

#### On Create (after successful RPC):
```typescript
writeAuditLog({
  action: 'create',
  resourceType: 'inventory_item',    // singular noun for the resource
  resourceId: data.id || '',          // the new record's ID
  resourceLabel: form.name,           // human-readable label
  details: { code: form.code }        // optional extra info
});
```

#### On Update (after successful RPC):
```typescript
// 1. Store original form as JSON string when edit starts (immune to Svelte 5 proxy)
let editFormOriginalJson = '';

// In startEdit():
editFormOriginalJson = JSON.stringify(editForm);

// 2. After successful update:
const originalForm = JSON.parse(editFormOriginalJson);
const currentForm = { name: editForm.name, price: editForm.price, ... };  // read each field explicitly
const changes = diffChanges(originalForm, currentForm, ['name', 'price', ...]);
writeAuditLog({
  action: 'update',
  resourceType: 'inventory_item',
  resourceId: itemId,
  resourceLabel: currentForm.name,
  changes
});
```

**CRITICAL:** Do NOT use `$state` for the original snapshot variable. Svelte 5's deep proxy will corrupt the comparison. Use a plain `let` with `JSON.stringify()` to store the original, then `JSON.parse()` to recover it before diffing.

#### On Delete (after successful RPC):
```typescript
writeAuditLog({
  action: 'delete',
  resourceType: 'inventory_item',
  resourceId: itemId,
  resourceLabel: itemName
});
```

#### On Status Change:
```typescript
writeAuditLog({
  action: 'status_change',
  resourceType: 'inventory_item',
  resourceId: itemId,
  resourceLabel: itemName,
  changes: [{ field: 'status', from: 'active', to: 'inactive' }]
});
```

### Integrating audit logs into an EXISTING window

When adding new operations to an existing window:

1. Import at the top (if not already):
   ```typescript
   import { writeAuditLog, diffChanges } from '$lib/utils/audit';
   ```

2. For new edit forms, add `let editFormOriginalJson = '';` (plain `let`, NOT `$state`).

3. Store the snapshot when edit starts:
   ```typescript
   editFormOriginalJson = JSON.stringify(formData);
   ```

4. After successful save, diff and log:
   ```typescript
   const original = JSON.parse(editFormOriginalJson);
   const current = { field1: editForm.field1, field2: editForm.field2 };
   const changes = diffChanges(original, current, ['field1', 'field2']);
   writeAuditLog({ action: 'update', resourceType: '...', resourceId: '...', resourceLabel: '...', changes });
   ```

### Available actions

| Action | When to use |
|--------|-------------|
| `create` | New record created |
| `update` | Record fields modified |
| `delete` | Record removed |
| `status_change` | Active/inactive toggle |
| `permission_change` | User permissions modified |
| `login` | User logs in (handled in TeamLoginPanel) |
| `logout` | User logs out (handled in desktop +page) |
| `view` | Window opened (handled automatically) |

### Common mistakes to avoid

1. **Never store the original form as `$state`** — Svelte 5's deep proxy will make both original and current always match.
2. **Never use `JSON.parse(JSON.stringify(editForm))` directly on a `$state` proxy for the original** — store it as a string in a plain `let` variable instead.
3. **Always read current form fields explicitly** — `{ name: editForm.name, ... }` — don't pass the proxy object directly to `diffChanges`.
4. **Always call `writeAuditLog` AFTER the success toast** — ensures we only log successful operations.
5. **The `changes` array format** — each entry must be `{ field: string, from?: any, to?: any }`.
6. **resourceType should be singular** — `'user'` not `'users'`, `'branch'` not `'branches'`.

## Opening Balance / Double-Entry Accounting System

When any entity (supplier, customer, bank account, cash account, etc.) is created with an **opening balance**, the system must create a proper double-entry transaction using the **Opening Balance Equity** ledger (`LED-OBE-000001`).

### How it works

- There is ONE system ledger: **Opening Balance Equity** (under `AG-CAPITAL` / Equity group)
- Ledger code: `LED-OBE-000001`
- When an opening balance > 0 is provided during entity creation, a `journal` type `accounting_transaction` is created with two `accounting_entries`

### Double-entry rules by entity type

| Entity | Entity Ledger Side | Opening Balance Equity Side |
|--------|-------------------|---------------------------|
| **Supplier** | Credit (you owe them) | Debit |
| **Customer** | Debit (they owe you) | Credit |
| **Cash/Bank** | Debit (your asset) | Credit |
| **Employee** | Credit (you owe salary) | Debit |
| **Loan Payable** | Credit (liability) | Debit |

### Reusable helper function

A reusable RPC function `rpc_create_opening_balance_entry` is available:

```typescript
await supabase.rpc('rpc_create_opening_balance_entry', {
  p_ledger_id: 'uuid-of-entity-ledger',
  p_amount: 50000,
  p_balance_side: 'credit',  // 'credit' for suppliers/liabilities, 'debit' for customers/assets
  p_description: 'Opening balance for Supplier XYZ'
});
```

This function:
1. Looks up the Opening Balance Equity ledger (`LED-OBE-000001`)
2. Gets the current financial year
3. Generates a voucher number (`OB-000001`, `OB-000002`, etc.)
4. Creates a `journal` transaction with two balanced entries
5. Returns `{ success, message, voucher_number }`

### When implementing NEW entity creation with opening balance

1. Add `opening_balance` field to the creation form (type `number`, default `0`)
2. Pass `p_opening_balance` to the RPC function
3. Inside the RPC:
   - If `p_opening_balance > 0`:
     - Get `v_obe_ledger_id` from `ledgers WHERE ledger_code = 'LED-OBE-000001'`
     - Get `v_fy_id` from `financial_years WHERE is_current = TRUE`
     - Generate voucher: `'OB-' || lpad(nextval('voucher_ob_seq')::TEXT, 6, '0')`
     - Insert `accounting_transactions` (voucher_type = 'journal', reference_type = 'opening_balance')
     - Insert two `accounting_entries`:
       - One for the entity ledger (debit or credit depending on type)
       - One for Opening Balance Equity (the opposite side)

4. Alternatively, after creating the entity and its ledger, call the helper:
   ```sql
   PERFORM rpc_create_opening_balance_entry(v_ledger_id, v_balance, 'credit', 'Opening balance for ' || entity_name);
   ```

### Already implemented for:
- **Suppliers** — `rpc_create_supplier` with `p_opening_balance` parameter (credit side)

### To implement for (when building these features):
- **Customers** — debit side (they owe you)
- **Cash accounts** — debit side (your asset)
- **Bank accounts** — debit side (your asset)
- **Employee payables** — credit side (you owe them)

### The Opening Balance Equity account should net to ZERO when all opening balances are correctly entered. A non-zero balance indicates the books are unbalanced (like Tally's "Difference in Opening Balances").
